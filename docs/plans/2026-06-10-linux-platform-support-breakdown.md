# Linux Platform Support Implementation Breakdown

**Goal:** Prepare implementation of Linux desktop support and the OS-neutral architecture described by ADR 0004 before changing production code.

**Architecture Decision:** `docs/adr/0004-linux-platform-support-and-os-neutral-architecture.md`

**Primary product constraint:** Agent terminal output must remain stable and continuous while users switch surfaces, split panes, restore sessions, and keep multiple agent conversations running side by side.

**Target baseline:** Ubuntu Desktop LTS packaged GUI app first. Fedora Workstation and other desktop environments are follow-up validation targets. Windows remains unsupported in this phase.

---

## Scope Guardrails

- Preserve current macOS behavior unless a change is explicitly listed as compatibility-reviewed.
- Keep `KMUX_SOCKET_PATH` as the macOS/Linux POSIX wire and env contract.
- Centralize platform behavior in explicit boundaries instead of spreading `process.platform` checks through terminal, pane, restore, rendering, or agent workflow code.
- Treat pty-host IPC as desktop-owned implementation detail, not a cross-package core contract.
- Keep CLI path resolution pure and side-effect-free.
- Keep socket runtime storage separate from state, data, cache, captures, attachments, raw output, and native extraction caches.
- Ship Linux stable only after shell spawn, packaged AppImage, updater, hooks, notifications, external sessions, usage, and output-continuity checks pass.

## Implementation Shape

Work should be split into milestone PRs. The first PRs should prove unknown Linux facts and move high-risk boundaries with characterization tests. Linux-specific behavior should be enabled only after those boundaries are test-covered.

Recommended milestone order:

1. Linux spike and fact log
2. Socket/path robustness shared by macOS and Linux
3. macOS behavior-preserving platform skeleton
4. `@kmux/persistence` path resolver expansion
5. Shell launch policy and hook runtime env
6. pty-host protocol ownership migration
7. Walking skeleton gate
8. Agent storage roots, credentials, subprocess, and watch policy
9. Renderer platform descriptor, keyboard policy, chrome, and typography
10. Linux packaging, updater, desktop identity, and notifications
11. Stable Linux release candidate validation

Each milestone should leave the app in a working state on macOS. Linux may be incomplete until the release candidate milestone, but incomplete Linux behavior must remain clearly gated.

## Review Boundaries

Prefer separate review boundaries for:

- Linux spike findings and architecture adjustments.
- Socket/path robustness.
- macOS behavior-preserving platform skeleton.
- Linux path resolver and storage separation.
- Shell launch policy and hook env.
- pty-host protocol ownership migration.
- Walking skeleton validation.
- Agent storage/credential/subprocess/watch policy.
- Renderer platform descriptor and keyboard policy.
- Packaging/updater/desktop identity.
- Release candidate validation documentation.

Do not mix pty-host protocol migration, renderer shortcut migration, and Linux packaging in the same review unless the change is strictly mechanical and already covered by tests.

- ADR 0004 has a corresponding implementation history with completed milestones.

- Linux stable build is not publicly shipped until the release candidate gate passes.
- Linux user and release documentation is updated before public stable publishing is enabled.
