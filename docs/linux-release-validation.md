# Linux Release Validation

This checklist is the evidence ledger for the Linux stable release candidate. Do not enable public Linux release publishing until every required Linux item is backed by current Ubuntu Desktop evidence and macOS compatibility checks remain green.

Current status: not passed. The current implementation has automated macOS/Darwin-host coverage, a passing macOS packaged release check, and gated Linux scripts, but the real Ubuntu Desktop AppImage, updater, notification identity, and compositor checks remain manual.

## Release Gate Commands

Run portable checks first:

```bash
npm run typecheck
npm run lint
npm test
npm run gate:walking-skeleton
```

```bash
npm run gate:walking-skeleton:linux
npm run package:linux
npm run smoke:packaged:linux
```

If a Linux target preflight prints `RC evidence: no on this host`, the current machine has not produced RC evidence. Stop treating that run as signoff, move the command to an Ubuntu Desktop LTS/AppImage target, or record the item as blocked/manual validation when that target is unavailable.

The generated `Git dirty:` handoff value is source-worktree provenance: generated release artifact directories such as `release-assets` and `apps/desktop/release` are ignored for that status, while normal modified or untracked source files still make it dirty.

The `System Tool Availability` section records `script` command availability, and the `Subprocess And Font Samples` section records a bounded `lsof -Pan -iTCP -sTCP:LISTEN` sample next to the `ps` process-table sample so subscription-probe, process-table, and listening-port metadata assumptions are visible during the Linux subprocess audit.

`gate:walking-skeleton:linux` also fails unless it runs on Ubuntu Desktop LTS with `DISPLAY` or `WAYLAND_DISPLAY` and Ubuntu/GNOME/Unity desktop session env. It must run the complete smoke, build, and selected Playwright checks; its output includes `Gate mode: Ubuntu Desktop Linux gate` and `RC evidence: walking-skeleton component only`, meaning it is one ledger component rather than a full RC pass. `--skip-e2e` and `--skip-build` are accepted only by the portable preflight and are rejected by the Linux desktop gate. Use `gate:walking-skeleton` only as a portable preflight; its output says `RC evidence: no` and does not satisfy the Ubuntu Desktop gate evidence rows.

`smoke:packaged:linux` fails unless it runs on Ubuntu Desktop LTS with `DISPLAY` or `WAYLAND_DISPLAY` and Ubuntu/GNOME/Unity desktop session env, then prints a preflight summary before the packaged Playwright smoke starts. The summary includes a `Smoke mode` line and `Passing RC evidence: no automatic pass`; include that output in the RC evidence because it records the selected `kmux-<version>-linux-<arch>.AppImage` artifact, arch and size, non-empty AppImage blockmap sidecar, Linux update metadata path/version/AppImage entry, update metadata top-level sha512, AppImage file-entry sha512, packaged AppImage sha512, update metadata AppImage size/checksum, packaged AppImage checksum match, extracted desktop-entry identity, packaged notification icon resource, AppImage runtime env used for the launch, and that `--no-sandbox` was not injected by the smoke wrapper. The summary also states that release visibility and updater check/download/install remain separate manual observations, and that notification delivery/window grouping remains a separate manual observation. If the AppImage only starts after a manual `--no-sandbox` retry, record that separately with the product/security decision; do not treat the default smoke as passed. `node scripts/smoke-packaged-linux.mjs --allow-any-linux-desktop` is only for non-RC diagnostics on other Linux desktops.

Run macOS compatibility checks on macOS:

```bash
npm run release:check:mac
```

`release:check:linux` intentionally runs only on Ubuntu Desktop LTS with `DISPLAY` or `WAYLAND_DISPLAY` and Ubuntu/GNOME/Unity desktop session env. It performs a fast desktop preflight, runs `gate:walking-skeleton:linux`, builds the AppImage, runs the packaged Linux smoke, and verifies that public GitHub release uploads remain gated while `KMUX_ENABLE_LINUX_PUBLIC_RELEASE` is unset. After `KMUX_ENABLE_LINUX_PUBLIC_RELEASE=1`, the public gate also requires a `kmux-<version>-linux-<arch>.AppImage`, matching AppImage blockmap sidecar, colocated `latest-linux*.yml` update metadata, no unexpected Linux release assets outside the current AppImage-only policy, an AppImage artifact upload path in the release workflow, and the gate before GitHub release publishing. The release workflow still calls `node scripts/release-check-linux.mjs` before any `gh release upload`, `gh release create`, `gh release edit`, or GitHub release action publishing step, without the desktop preflight or walking-skeleton gate, as a lightweight public-publishing gate for already-built macOS assets.

## Public Publishing Gate

Linux public publishing stays disabled until this document records a passing stable RC run.

Before that point:

- do not set `KMUX_ENABLE_LINUX_PUBLIC_RELEASE=1` in public release jobs
- keep `package:linux` using `--publish never`
- do not add workflow `electron-builder --linux` commands unless they include `--publish never`, and do not add workflow `npm run dist:linux` commands with public `--publish` values such as `always`, `onTag`, or `onTagOrDraft`
- do not add public release artifact names or download patterns that include both `linux` and `release-assets`, or broad release-asset artifact patterns such as `*-release-assets`
